## Locked-in design decisions (don't relitigate)

Most of these are inherited from PR1–4 + the spec/forms.md
preview; relitigating would invalidate already-shipped work.

- **Intrusive-hashmap shape.** Value type S carries its own key
  as one of its fields named by `indexed_by`. `set(value: S)`
  takes the whole struct; the substrate extracts the key at
  insert. No `Map<K, V>` shape; no separate (K, V) pairs.
- **Pool slot kind.** The slot is `pool entries of S indexed_by
  K;` — Pool, not Heap. Hashmap recycles cells as entries come
  and go; that IS the pool discipline.
- **Method surface.** `get(K) -> S fallible(KeyError)`,
  `set(S) -> ()`, `has(K) -> Bool`, `remove(K) -> ()
  fallible(KeyError)`, `len() -> Int`, `is_empty() -> Bool`.
  Locked in PR3.
- **Key types at v1.** Int and String only. The C runtime
  switches on a `key_type_tag` field (`LOTUS_HASHMAP_KEY_INT =
  0`, `LOTUS_HASHMAP_KEY_STRING = 1`). Codegen picks the tag
  from the indexed-by field's resolved type.
- **`KeyError` shape.** `{ kind: String }`. Minimal. The key
  isn't carried because K varies per hashmap; users wanting key
  context construct it via `or <fallback>` substitution.
- **No tombstones.** Backward-shift deletion in the C runtime.
  Probe chains stay tight.
- **Cap is power-of-two.** Hash-to-index is `& mask`. Initial
  cap 8, double at 0.7 load factor.

## What's missing (PR5 / PR6 / PR7)

### PR5 — codegen for `@form(hashmap)`

The substantive piece. Mirror the `@form(vec)` codegen pattern,
which is established and tested. Key entry points:

- **`crates/hale-codegen/src/codegen.rs:1230`** — `enum
  SlotForm { Vec, }`. Add `Hashmap` variant.
- **`crates/hale-codegen/src/codegen.rs:1492`** — extern fn
  declarations for `lotus_vec_*`. Add a parallel block for
  `lotus_hashmap_*` declaring all 8 fns with the C signatures
  from PR4 (`lotus_arena.c`). Pattern:
  ```
  void  lotus_hashmap_init(ptr, i64 key_size, i64 value_size, i32 key_type_tag)
  void  lotus_hashmap_set(ptr, ptr key, ptr value)
  i32   lotus_hashmap_get(ptr, ptr key, ptr out_value)
  i32   lotus_hashmap_has(ptr, ptr key)
  i32   lotus_hashmap_remove(ptr, ptr key)
  i64   lotus_hashmap_len(ptr)
  i32   lotus_hashmap_is_empty(ptr)
  void  lotus_hashmap_destroy(ptr)
  ```
- **`crates/hale-codegen/src/codegen.rs:6526`** — where
  `slot.form` gets set to `Some(SlotForm::Vec)` at form-locus
  layout time. Add the parallel branch setting
  `Some(SlotForm::Hashmap)` when the locus has
  `@form(hashmap)`. The slot's inline storage is the
  `lotus_hashmap_t` struct rather than `lotus_vec_t`.
- **`crates/hale-codegen/src/codegen.rs:19017`** — locus
  birth emits the slot init call. `SlotForm::Vec` calls
  `lotus_vec_init`. Add `SlotForm::Hashmap` branch calling
  `lotus_hashmap_init` with `key_size`, `value_size`,
  `key_type_tag` arguments. Key size + tag come from the
  indexed-by field's resolved type at codegen-time.
- **`crates/hale-codegen/src/codegen.rs:20977`** —
  `try_lower_form_vec_method` is the synth-method dispatcher.
  Add a parallel `try_lower_form_hashmap_method` returning
  the same `Result<Option<Option<...>>, _>` shape. Hook it
  into the two callers at lines `20277` and `20637` (the
  method-call lowering sites).
- **`crates/hale-codegen/src/codegen.rs:9776`** —
  `try_lower_form_vec_fallible_method` handles `get`/`pop`
  (the fallible-int-returning vec methods). Add a parallel
  `try_lower_form_hashmap_fallible_method` for `get` and
  `remove` (the hashmap fallible methods). Hook it at line
  `9745`.
- **`crates/hale-codegen/src/codegen.rs:23791`** — locus
  dissolve emits the slot destroy call. `SlotForm::Vec` calls
  `lotus_vec_destroy`. Add `SlotForm::Hashmap` branch calling
  `lotus_hashmap_destroy`.

The **key extraction** at method-call sites is the new piece
the vec-codegen doesn't have. At `set(value: S)`:
- Lower `value` to a struct alloca.
- GEP the indexed-by field by offset (the field index is known
  from the type-resolver pass).
- Pass `(slot_ptr, key_field_ptr, value_alloca_ptr)` to
  `lotus_hashmap_set`.

At `get(key: K)`, `has(key: K)`, `remove(key: K)`:
- Lower the key arg into an alloca matching `key_size`.
- Pass `(slot_ptr, key_alloca_ptr, ...)` to the appropriate
  runtime fn.

the test template. Write a parallel `form_hashmap_codegen.rs`

- `hashmap_int_keyed_set_and_get`

## Recommended first move

Read the codegen.rs sections named above for `@form(vec)` —
specifically `try_lower_form_vec_method` (around line 20977)
and `try_lower_form_vec_fallible_method` (around line 9776).
Those are ~150-line and ~250-line functions respectively that
the hashmap variants mirror almost exactly. The differences are:

1. Method names + arg counts (get/set/has/remove/len/is_empty
   vs push/get/pop/len/is_empty).
2. Key extraction at the set call site (GEP the indexed-by
   field).
3. Key materialization at get/has/remove sites (lower the key
   arg into an alloca).
4. Fallible-int handling for `get` and `remove` (both fallible
   in hashmap; only `get` and `pop` were fallible in vec).

Spend the first 30 minutes reading, then start with PR5's
smallest slice: add the `SlotForm::Hashmap` variant, declare
the extern fns, wire up init + destroy. Run `cargo build` to
confirm no breakage. Then add the method dispatcher with one
infallible method (`len()` or `is_empty()`) and verify
end-to-end before adding `set`/`has` (infallible) and finally
`get`/`remove` (fallible).

## What's deferred (don't expand scope)

- **Iteration surface.** No `for k, v in hashmap { ... }` at
  v1; users walk via external bookkeeping or a future
  iteration API.
- **Generic / custom hash functions.** v1 ships Int + String
  only with fixed hashers. Adding user-defined Hash trait or
  custom-key types waits on a workload.
- **Capacity hints.** `@form(hashmap, cap = 64)` is rejected
  in PR2; no tuning knobs at v1. Workload-driven.
- **Set type.** A `@form(set)` would be a hashmap-without-
  value variant. Not part of FORM-4; revisit if a workload
  needs it.
- **Bench harness.** v1.x-FORM-3 was the within-10%-of-C bench
  gate for `@form(vec)`. An equivalent for `@form(hashmap)` is
  a future milestone after PR5–7 ship the surface.
